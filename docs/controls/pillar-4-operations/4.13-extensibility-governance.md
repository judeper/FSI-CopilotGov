# Control 4.13: Copilot Extensibility Governance (Plugin Lifecycle, Connector Monitoring)

**Control ID:** 4.13
**Pillar:** Operations & Monitoring
**Regulatory Reference:** GLBA 501(b), FFIEC IT Examination Handbook, SOX Section 404, OCC Third-Party Risk Management Guidance
**Last Verified:** 2026-02-17
**Governance Levels:** Baseline / Recommended / Regulated

---

## Objective

Establish ongoing governance controls for deployed Copilot extensibility components -- including plugins, Microsoft Graph connectors, and declarative agents -- covering the full lifecycle from approval through deployment, monitoring, security re-assessment, and eventual decommission, to support compliance with third-party risk management expectations and data protection requirements in financial services environments.

## Why This Matters for FSI

Copilot extensibility features allow organizations to expand Copilot's capabilities beyond native Microsoft 365 data by connecting external data sources (Graph connectors), integrating third-party services (plugins), and creating custom AI experiences (declarative agents). While Control 1.13 addresses pre-deployment readiness and Control 2.13 addresses security governance, this control focuses on the ongoing operational governance of these components after deployment.

For financial institutions, extensibility governance is critical because:

**GLBA 501(b):** Graph connectors that ingest customer data from external systems (CRM, loan origination, account management) into the Microsoft 365 search and Copilot grounding index create new data flows that must be protected under GLBA safeguard requirements. Ongoing monitoring of these connectors helps verify that data protection controls remain effective.

**FFIEC IT Examination Handbook:** The FFIEC expects institutions to monitor third-party integrations throughout their lifecycle, not just at initial deployment. Plugins that connect Copilot to external services represent ongoing third-party dependencies that require periodic assessment.

**SOX Section 404:** If extensibility components are used in financial reporting workflows (e.g., a Graph connector that brings financial data into Copilot's grounding scope, or a plugin that integrates with the ERP), they become part of the ITGC scope and require change management, access controls, and monitoring.

**OCC Third-Party Risk Management:** The OCC expects banks to manage third-party relationships throughout the lifecycle, including ongoing monitoring. Plugins and connectors that integrate third-party services into Copilot represent third-party technology dependencies subject to this guidance.

The operational phase introduces risks that do not exist during initial deployment: configuration drift, credential expiration, data source changes, vendor updates, and accumulated permission sprawl. Without ongoing governance, deployed extensibility components may degrade in security posture over time.

## Control Description

**Extensibility Component Inventory**

Maintain a comprehensive inventory of all deployed Copilot extensibility components:

| Component Type | Inventory Fields |
|---------------|-----------------|
| **Graph Connectors** | Connector name, data source, data types ingested, owner, deployment date, last security review, connection status |
| **Plugins (Message Extensions)** | Plugin name, vendor, API endpoints, data access scope, owner, deployment date, last security review, user access |
| **Declarative Agents** | Agent name, purpose, data sources, creator, deployment date, sharing scope, last review |
| **Custom Connectors** | Connector type, data source, authentication method, owner, deployment date, credential expiration |

**Lifecycle Management Stages**

Each extensibility component should progress through a governed lifecycle:

| Stage | Activities | Owner |
|-------|-----------|-------|
| **Request** | Business justification, data flow documentation, initial risk assessment | Business requestor + IT |
| **Approval** | Security review, compliance review, architecture review | CAB / Governance committee |
| **Deployment** | Technical implementation, access configuration, monitoring setup | IT Operations |
| **Monitoring** | Usage tracking, error monitoring, data flow verification | IT Operations |
| **Re-Assessment** | Periodic security review, access audit, vendor assessment | Security + Compliance |
| **Update** | Version updates, configuration changes, scope modifications | IT Operations (with approval) |
| **Decommission** | Access removal, data cleanup, inventory update, documentation archive | IT Operations |

**Usage Monitoring**

Monitor deployed extensibility components for health and governance:

| Metric | Description | Alert Threshold |
|--------|-------------|-----------------|
| Active Users | Number of users interacting with the component | Unexpected drop (component may be broken) or spike (unauthorized access) |
| Data Volume | Volume of data ingested (connectors) or exchanged (plugins) | Significant increase may indicate scope creep |
| Error Rate | Failed interactions or connection errors | >5% error rate triggers investigation |
| Latency | Response time for plugin calls or connector sync | Degradation may indicate infrastructure issues |
| Authentication Failures | Failed authentication attempts | Any failures trigger immediate investigation |
| Data Freshness | Last successful data sync for connectors | Stale data may indicate broken connections |

**Security Re-Assessment Cadence**

| Component Risk Level | Assessment Cadence | Assessment Scope |
|---------------------|-------------------|-----------------|
| High (processes customer PII, financial data) | Quarterly | Full security review, penetration test results, vendor SOC report |
| Medium (processes internal business data) | Semi-annually | Security configuration review, access audit, vendor assessment |
| Low (processes non-sensitive data) | Annually | Configuration review, access verification |

**Decommission Procedures**

When an extensibility component is no longer needed:

| Step | Action | Verification |
|------|--------|-------------|
| 1 | Disable the component (do not delete immediately) | Users can no longer invoke the component |
| 2 | Revoke API credentials and service principal permissions | Authentication no longer possible |
| 3 | Remove ingested data from search index (Graph connectors) | Content no longer appears in Copilot responses |
| 4 | Update the component inventory | Inventory reflects decommissioned status |
| 5 | Archive configuration and approval documentation | Records retained per retention schedule |
| 6 | Notify stakeholders of decommission | Business users aware of change |
| 7 | Delete component after verification period (30 days) | Complete removal from environment |

## Copilot Surface Coverage

| Surface | Extensibility Type | Governance Priority |
|---------|-------------------|-------------------|
| M365 Business Chat | Graph connectors (grounding), Plugins (actions), Declarative Agents | Critical -- cross-app data access |
| Teams Copilot | Plugins (message extensions), Declarative Agents | High -- collaboration context |
| Outlook Copilot | Plugins (email actions) | Medium -- communication context |
| Word / Excel / PowerPoint | Plugins (data retrieval, actions) | Medium -- document context |
| SharePoint Copilot | Graph connectors (content indexing), Declarative Agents from SharePoint | High -- data grounding scope |
| Copilot Studio | Custom agents with organizational data | Critical -- custom AI behavior |

## Governance Levels

### Baseline

- [ ] Maintain a current inventory of all deployed Copilot extensibility components
- [ ] Assign an owner to each deployed component responsible for ongoing governance
- [ ] Monitor component health (connection status, error rates) at least monthly
- [ ] Establish a decommission process for extensibility components no longer needed
- [ ] Review and renew API credentials and certificates before expiration
- [ ] Document data flows for each Graph connector and plugin

### Recommended

- [ ] Implement automated monitoring for component health and usage metrics
- [ ] Conduct security re-assessments per the risk-based cadence defined in this control
- [ ] Integrate extensibility component changes into the institution's change management process (Control 4.12)
- [ ] Review user access to each component quarterly and remove unnecessary permissions
- [ ] Track vendor SOC reports and security assessments for third-party plugin providers
- [ ] Implement automated alerts for authentication failures, data freshness issues, and error rate spikes
- [ ] Conduct annual review of the entire extensibility inventory with business stakeholders

### Regulated

- [ ] Include extensibility components in the institution's third-party risk management program
- [ ] Require vendor SOC 2 Type II reports for all third-party plugins processing regulated data
- [ ] Include extensibility governance in SOX ITGC scope for components used in financial workflows
- [ ] Conduct annual penetration testing of externally-facing plugin API endpoints
- [ ] Present extensibility governance status to the technology risk committee quarterly
- [ ] Implement DLP monitoring for data flowing through extensibility components
- [ ] Maintain 7-year records of extensibility component approvals, assessments, and decommissions
- [ ] Engage internal audit to review extensibility governance effectiveness annually

## Setup & Configuration

### Step 1: Build Component Inventory

1. Navigate to **M365 Admin Center > Settings > Integrated Apps** to inventory deployed plugins
2. Navigate to **Microsoft Search Admin Center > Data Sources > Connectors** for Graph connectors
3. Review **Copilot Studio** for deployed declarative agents
4. Review **Teams Admin Center > Teams Apps** for Teams-integrated plugins
5. Compile inventory in a structured format:

```
Component: [Name]
Type: Graph Connector / Plugin / Declarative Agent
Vendor: [Vendor name or "Internal"]
Data Sources: [What data it accesses]
Data Classification: [Sensitivity level of data involved]
Owner: [Name and department]
Deployment Date: [Date]
Last Security Review: [Date]
Next Review Due: [Date]
Credential Expiration: [Date]
Status: Active / Under Review / Scheduled for Decommission
```

### Step 2: Configure Monitoring

For each component, establish monitoring:

**Graph Connectors:**
1. Navigate to **Microsoft Search Admin Center > Data Sources > Connectors**
2. Monitor connection status and last crawl time
3. Set up alerts for crawl failures or data freshness issues
4. Track items indexed vs. expected to detect data loss

**Plugins:**
1. Monitor through **M365 Admin Center > Settings > Integrated Apps**
2. Track usage via admin reports
3. Monitor API endpoint health (if externally hosted)
4. Track authentication token usage and expiration

**Declarative Agents:**
1. Monitor through **Copilot Studio** analytics
2. Track conversation volumes and error rates
3. Review data source connections for health
4. Monitor user feedback on agent responses

### Step 3: Establish Re-Assessment Process

Create a re-assessment checklist:

| Assessment Area | Review Items |
|----------------|-------------|
| **Access Review** | Who can use the component? Is access still appropriate? |
| **Data Scope** | Has the data scope changed since last review? Is data classification still accurate? |
| **Security Configuration** | Are credentials current? Is encryption in place? Are API endpoints secured? |
| **Vendor Assessment** | Is the vendor's SOC report current? Any security incidents reported? |
| **Business Justification** | Is the component still needed? Has the business case changed? |
| **Compliance Impact** | Any regulatory changes that affect this component? |
| **Performance** | Is the component performing within expected parameters? |

### Step 4: Implement Credential Management

1. Inventory all credentials used by extensibility components:
   - API keys, OAuth tokens, certificates, service principal secrets
2. Track expiration dates in a centralized credential management system
3. Configure alerts 60 days before credential expiration
4. Establish credential rotation procedures:
   - Generate new credentials before old ones expire
   - Update component configuration with new credentials
   - Verify functionality after rotation
   - Revoke old credentials after confirmation

### Step 5: Configure Data Flow Monitoring

For Graph connectors and plugins that process sensitive data:

1. Implement DLP policies that monitor data flowing through extensibility channels
2. Configure Sentinel analytics rules for extensibility-related security events (see Control 4.11)
3. Establish data volume baselines and alert on significant deviations
4. Monitor for data leakage indicators:
   - Unusual data volume increases
   - New data types appearing in the index
   - Unexpected external API calls

## Financial Sector Considerations

**Third-Party Risk Management:** Plugins and connectors from third-party vendors represent third-party technology relationships. For banks subject to OCC guidance, these relationships should be included in the third-party risk management program. This includes initial due diligence, ongoing monitoring, and periodic re-assessment. The depth of oversight should be proportional to the criticality and data sensitivity of the component.

**Data Aggregation Risk:** Graph connectors that bring data from multiple source systems into the Copilot grounding index may create a data aggregation point that is more sensitive than any individual source. For example, connecting both a CRM and a loan origination system to Copilot creates a combined data set that could be used for unauthorized purposes. Assess aggregation risk during re-assessments.

**Regulatory Data in Scope:** If extensibility components process data subject to specific regulatory requirements (e.g., SEC books and records, FINRA communications, HIPAA-protected health information for insurance), verify that the component's data handling meets those specific requirements. A Graph connector that ingests trade data into Copilot makes that data subject to the same retention and access controls as the source system.

**Vendor Concentration:** Multiple Copilot plugins from the same vendor create vendor concentration risk. If the vendor experiences a service disruption or security incident, multiple Copilot capabilities may be affected simultaneously. Assess vendor concentration as part of the extensibility governance program.

**Compliance with Custom Agents:** Declarative agents created from SharePoint sites can expose organizational content through custom conversational experiences. If these agents are configured to access regulated content, they should be subject to the same compliance controls as direct access to that content, including DLP, retention, and supervisory review where applicable.

**SOC Report Requirements:** For regulated environments, request SOC 2 Type II reports from third-party plugin vendors. Review the report for relevant trust service criteria (security, availability, processing integrity, confidentiality, privacy). Document the review and any noted exceptions.

## Verification Criteria

| # | Verification Step | Expected Result |
|---|------------------|-----------------|
| 1 | Review extensibility component inventory | Inventory current and complete with all required fields |
| 2 | Verify each component has an assigned owner | All components have designated owners |
| 3 | Confirm monitoring is active for all components | Health monitoring configured and generating data |
| 4 | Review security re-assessment schedule | All components have re-assessment dates within cadence |
| 5 | Verify credential expiration tracking | All credentials tracked with expiration alerts configured |
| 6 | Confirm decommission process documentation | Decommission procedures documented and tested |
| 7 | Review vendor SOC reports for third-party components (Regulated) | Current SOC 2 Type II reports on file for all critical vendors |
| 8 | Verify data flow documentation for Graph connectors | Data flow diagrams current and reflecting actual configuration |
