# Control 2.14: Declarative Agents from SharePoint — Creation and Sharing Governance

**Control ID:** 2.14
**Pillar:** Security & Protection
**Regulatory Reference:** GLBA 501(b)
**Last Verified:** 2026-02-17
**Governance Levels:** Baseline / Recommended / Regulated

---

## Objective

Govern the creation, sharing, and data boundary enforcement for declarative agents built from SharePoint sites. SharePoint declarative agents allow site owners to create Copilot agents scoped to specific SharePoint site content, providing focused AI assistance for particular topics or workflows. In financial services, these agents must be governed to prevent uncontrolled agent proliferation, data boundary violations, and unauthorized access to sensitive site content through AI interactions. This control defines who can create agents, how agents are shared, and what data agents can access.

---

## Why This Matters for FSI

- **GLBA 501(b)** requires safeguards for customer information — declarative agents that access sites containing customer data must be governed to prevent unauthorized AI-assisted access to that data
- **Data boundary enforcement** is essential when agents are scoped to specific sites — agents should not be able to access content beyond their defined scope, and users should not be able to circumvent site permissions through agent interactions
- **Agent proliferation risk** is a governance concern — without controls, any site owner could create agents that expose site content to AI interactions, potentially broadening the data exposure surface
- **FFIEC expectations for IT governance** extend to AI tool management — the creation of custom AI agents should follow established IT governance processes
- **OCC/Fed Interagency AI Guidance** expects institutions to maintain oversight of AI capabilities deployed within the organization, including custom configurations

---

## Control Description

SharePoint declarative agents are Copilot agents configured to ground their responses on content from specific SharePoint sites. Unlike full-code Copilot Studio agents, declarative agents are created through a no-code interface by SharePoint site owners.

### Declarative Agent Architecture

```
Site Owner creates Agent
         │
         ▼
┌─────────────────────────┐
│  SharePoint Declarative  │
│  Agent Configuration     │
│                         │
│  ├─ Agent name/desc     │
│  ├─ Grounding scope     │
│  │   (specific sites)   │
│  ├─ Instructions        │
│  ├─ Welcome message     │
│  ├─ Web search (on/off) │
│  └─ Sharing settings    │
└────────────┬────────────┘
             │
      ┌──────┴──────┐
      │             │
   Private       Shared
   (creator)     (org/group)
                    │
             Users interact
             with agent
                    │
             Agent grounds on
             scoped SharePoint
             content only
```

### Agent Creation and Sharing Controls

| Control | Description | Admin Configuration |
|---------|-------------|-------------------|
| **Who can create agents** | Controls which users can create declarative agents | M365 Admin Center > Copilot settings |
| **Agent sharing scope** | Controls who agents can be shared with | SharePoint Admin Center + agent settings |
| **Data boundary** | Limits agent grounding to specified sites | Agent configuration (site selection) |
| **Web search toggle** | Whether agent can use web search | Per-agent setting |
| **Instructions governance** | System instructions that guide agent behavior | Agent creator defines; admin can audit |
| **Agent catalog** | Visibility of agents to other users | Controlled deployment vs. org-wide |

### Agent Risk Assessment Matrix

| Risk Factor | Low Risk | Medium Risk | High Risk |
|-------------|----------|-------------|-----------|
| **Grounding scope** | Public intranet content | Departmental business content | Client data, regulatory content |
| **Sharing scope** | Creator only | Specific team members | Org-wide or external |
| **Web search** | Disabled | N/A | Enabled |
| **Instructions** | General purpose | Role-specific | Accesses sensitive workflows |
| **Creator** | IT-managed | Business owner (trained) | Any site owner (untrained) |

### Agent Data Boundary Enforcement

| Boundary Control | How It Works | Limitation |
|-----------------|-------------|------------|
| **Site scoping** | Agent grounded on selected sites only | Agent cannot access content outside selected sites |
| **Permission inheritance** | Users can only access content they have permissions for | Agent respects underlying SharePoint permissions |
| **Label enforcement** | DLP and label policies apply to agent interactions | Agent cannot bypass sensitivity label protections |
| **Information barriers** | IB policies apply to agent interactions | Agent respects organizational barriers |
| **RSS interaction** | If RSS is enabled, agent respects RSS boundaries | Agent grounding sites must be on RSS allow list |

### Agent Governance Matrix for FSI

| Agent Category | Creation Permitted | Approval Required | Sharing Scope | Review Cadence |
|---------------|:-:|:-:|---|---|
| HR/People agents | IT only | Yes | HR team | Quarterly |
| IT help desk agents | IT only | Yes | All employees | Quarterly |
| Department knowledge agents | Site owners (trained) | Business approval | Department | Semi-annually |
| Client-facing process agents | IT only | Compliance + IT | Specific roles | Quarterly |
| Compliance/regulatory agents | Compliance IT | Compliance approval | Compliance team | Monthly |
| General intranet agents | Site owners (trained) | No | All employees | Annually |

## Governance Levels

| Level | Requirement | Rationale |
|-------|-------------|-----------|
| **Baseline** | Restrict agent creation to IT administrators only; disable web search for all agents; all agents scoped to non-sensitive sites; agent inventory maintained; disable sharing of agents beyond the creating team | Maximum control during initial deployment — prevents agent proliferation and limits data exposure |
| **Recommended** | Extend agent creation to trained site owners with business approval; agent creation requires documented business justification; web search disabled by default (opt-in with approval); agents can be shared within the organization; quarterly agent inventory review; agent usage monitoring; agents on sites with sensitive content require compliance review | Controlled agent ecosystem with approval governance — allows business-led agent creation within guardrails |
| **Regulated** | All Recommended requirements plus: agent creation requires IT + compliance dual approval; all agents registered in a central governance catalog; agent instructions reviewed by compliance; monthly agent access and usage audit; agents on client data sites require annual security review; agent configuration changes require change management process; agent governance included in examination packages | Full agent governance with continuous oversight — designed for firms requiring comprehensive AI governance documentation |

## Setup & Configuration

### Step 1: Configure Agent Creation Permissions

**Portal:** Microsoft 365 Admin Center > Settings > Copilot > Agents

1. Navigate to agent creation settings
2. Set creation permissions based on governance level:
   - Baseline: IT administrators only
   - Recommended: IT + approved site owners
   - Regulated: IT + approved site owners with dual approval
3. Configure group-based creation permissions

### Step 2: Configure Default Agent Settings

1. Set default web search to **Off** for all new agents
2. Set default sharing scope to **Creator only**
3. Configure default data boundary to respect RSS settings

### Step 3: Establish Agent Approval Workflow

For Recommended and Regulated levels, implement an approval workflow:

| Step | Owner | Deliverable |
|------|-------|-------------|
| 1. Agent request | Business owner | Business justification + scope definition |
| 2. Data sensitivity review | Information Security | Assessment of grounding site sensitivity |
| 3. Compliance review (if required) | Compliance | Regulatory risk assessment for client data agents |
| 4. Approval | Agent governance committee | Approval/denial with conditions |
| 5. Creation | IT or approved site owner | Agent configuration per approved specifications |
| 6. Testing | Business owner + IT | Functional and security testing |
| 7. Deployment | IT | Controlled sharing to approved audience |

### Step 4: Configure Agent Monitoring

```powershell
# Search audit logs for agent-related activities
Search-UnifiedAuditLog -StartDate (Get-Date).AddDays(-30) -EndDate (Get-Date) `
  -Operations "CopilotAgentCreated","CopilotAgentShared","CopilotAgentInteraction" `
  -ResultSize 5000

# List all declarative agents (via Graph API)
# Note: Specific API endpoints may vary — check current Microsoft documentation
```

### Step 5: Build Agent Inventory

Maintain a central inventory of all declarative agents:

| Field | Description |
|-------|-------------|
| Agent name | Display name of the agent |
| Agent ID | Unique identifier |
| Creator | User who created the agent |
| Business owner | Responsible business stakeholder |
| Grounding sites | SharePoint sites the agent can access |
| Data sensitivity | Classification of grounding site content |
| Sharing scope | Who the agent is shared with |
| Web search | Enabled or disabled |
| Created date | Date of agent creation |
| Last reviewed | Date of last governance review |
| Approval status | Approved / pending / revoked |

### Step 6: Ongoing Governance

1. **Quarterly inventory review:** Verify all agents are documented and approved
2. **Usage analysis:** Review agent interaction volumes and patterns
3. **Orphan agent detection:** Identify agents whose creators have left the organization
4. **Scope creep monitoring:** Check whether agent grounding sites have changed since approval
5. **Stale agent cleanup:** Decommission agents with no usage in 90+ days

## Financial Sector Considerations

- **Agent Proliferation Risk:** Without governance, site owners could create hundreds of agents across the organization, each potentially exposing site content to AI interactions. FSI firms should treat agent creation like application deployment — requiring approval and oversight.
- **Client Data Agents:** Agents grounded on sites containing client data (portfolios, account information, transaction histories) are high-risk and require the strictest governance. Consider whether client data agents are appropriate at all, or whether standard Copilot with DLP is sufficient.
- **Instructions as Policy:** Agent instructions (the system prompt that guides agent behavior) can include guardrails such as "Do not provide investment advice" or "Always include compliance disclaimers." However, these instructions are not tamper-proof and should not be relied upon as regulatory controls.
- **Regulatory Examination:** If regulators ask about AI agent deployment, a comprehensive agent inventory with approval records, data sensitivity assessments, and usage metrics demonstrates sound governance.
- **Departing Employee Agents:** When site owners or agent creators leave the organization, their agents must be transferred to new owners or decommissioned. Include agent ownership transfer in offboarding procedures.
- **Agent as Communication Channel:** If agents are used by registered representatives to respond to client inquiries, agent interactions may constitute firm communications under FINRA rules and require supervision and retention (see Pillar 3 controls).
- **Testing Before Deployment:** Before sharing agents with users, test them with representative queries to verify they do not surface inappropriate content, provide investment advice, or violate information barriers.

---

## Verification Criteria

1. **Creation Permissions:** Verify that agent creation is restricted to authorized users — attempt to create an agent as an unauthorized user and confirm it is blocked
2. **Default Settings:** Verify that new agents default to web search disabled and creator-only sharing
3. **Agent Inventory:** Confirm a central agent inventory exists and is current (all deployed agents documented)
4. **Approval Records:** For Recommended/Regulated levels, verify that approved agents have documented approval records
5. **Data Boundary Enforcement:** Query an agent about content outside its scoped sites — verify the agent cannot access non-scoped content
6. **Permission Inheritance:** Access an agent as a user without permissions to the grounding sites — verify the agent respects underlying SharePoint permissions
7. **Web Search Control:** Verify that agents without web search approval cannot access web content
8. **Sharing Scope:** Verify that agent sharing respects the approved sharing scope (not shared more broadly than approved)
9. **Audit Logging:** Verify that agent creation, modification, sharing, and interaction events appear in audit logs
10. **Review Cadence:** Confirm that agent governance reviews are scheduled at the frequency required by the governance level
