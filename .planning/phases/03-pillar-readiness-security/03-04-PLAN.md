    **Control 2.1 updates (P2-01) — MAJOR EXPANSION:**

    **Dual DLP policy types — this likely warrants a new subsection given 3+ paragraphs of genuinely distinct conceptual content per user decision:**

    The current 2.1 documents only label-based response blocking. Add the full dual-type architecture:

    **DLP Policy Type 1: Label-Based Response Blocking**
    - Configured in the **Microsoft 365 Copilot** DLP policy location
    - When a user's Copilot prompt causes grounding against a file or email carrying a sensitivity label matching the policy condition, Copilot is blocked from including that content in its response
    - **Enforcement point:** Copilot's response generation (grounding phase)
    - **What is scanned:** Files and emails referenced during Copilot's retrieval
    - **Primary use:** Prevent Copilot from surfacing labeled content to users who should not receive it via AI
    - This is likely what the existing 2.1 already documents — update to clearly label it as "Type 1" or "Label-Based"

    **DLP Policy Type 2: SIT-Based Prompt Blocking**
    - A distinct policy type that scans the user's **prompt itself** for sensitive information types before Copilot processes the request
    - When a user types sensitive data directly into a Copilot prompt (e.g., pastes credit card numbers, SSNs), Copilot is blocked from responding, including blocking grounding via Microsoft Graph or web search
    - **Enforcement point:** The user's prompt (before Copilot processes it)
    - **What is scanned:** The text the user types into Copilot
    - **Primary use:** Prevent users from inadvertently submitting sensitive data to Copilot
    - These two policy types **cannot be merged into a single policy** — they are configured separately and address different risk vectors
    - Add FSI framing: per SEC Regulation S-P (17 CFR Section 248), amended effective December 3, 2025 for larger entities, customer information safeguards must cover AI interaction surfaces

    **Comparison table:**

    | Aspect | Label-Based Response Blocking | SIT-Based Prompt Blocking |
    |--------|------|------|
    | Enforcement point | Copilot response (grounding) | User prompt (input) |
    | What is scanned | Referenced files/emails | User's typed prompt |
    | Trigger | Sensitivity label on source content | SIT pattern match in prompt text |
    | Can be combined | No — separate policy required | No — separate policy required |
    | FSI use | Block AI surfacing of labeled NPI/MNPI | Block users pasting account numbers, SSNs into prompts |

    **Default DLP policy for Copilot:**
    - Document the default DLP policy that Microsoft deploys in **simulation mode** (GA January 2026, MC1182689). This is the SIT-based prompt blocking type.
    - The default policy detects sensitive information in Copilot prompts and runs in simulation mode (audit only, no blocking).
    - Accessible from: MAC > Copilot > Overview > Security tab OR Microsoft Purview > Data loss prevention > Policies.
    - Tier recommendations: Baseline — review default policy in simulation mode, enable at least the SIT-based policy; Recommended — enable both policy types, tune default policy from simulation to enforcement; Regulated — both types enforced with custom FSI SITs (ABA routing, CUSIP, ISIN, etc.), no simulation mode.

    **Edge browser DLP (GA September 2025):**
    - Add Edge browser DLP as an additional enforcement surface: DLP policies now extend to Copilot interactions within Microsoft Edge, preventing sensitive data from being submitted through browser-based Copilot interfaces.
    - This complements the native M365 app DLP — Edge DLP catches browser-based interactions that the native location may not cover.
    - Add to tier recommendations: Recommended — enforce Edge DLP; Regulated — Edge DLP mandatory with Endpoint DLP for complete coverage.

    **Update the existing architecture diagram** to show both policy types as distinct inspection points.

    **All 2.1 playbooks:**
    - Update portal-walkthrough with both DLP policy creation paths (Purview > DLP > Create policy selecting Copilot location for each type; MAC > Copilot > Security tab for default policy access).
    - Update PowerShell playbook with any applicable PowerShell configuration for the new policy types.
    - Update verification-testing with verification for both policy types and default policy.
    - Update troubleshooting with common issues (e.g., why label-based and SIT-based can't be combined, how to test in simulation mode).

    **Control 2.2 updates (P2-02):**

    **Label groups replacing parent labels:**
    - Document the migration from parent/child label hierarchy to **label groups** (GA December 2025 – January 2026, MC1111778).
    - Label groups provide a new organizational container that replaces the parent label concept. Migration is not immediate deletion — parent labels are being migrated to the label groups model.
    - Use exact terminology: "Parent labels migrating to label groups (GA January 2026)" — not "parent labels deprecated."
    - Document migration path: Microsoft Purview > Information Protection > Labels > Migrate sensitivity label scheme.
    - Add impact on Copilot: label groups affect how DLP policies reference label hierarchies and how Copilot evaluates label-based access decisions.
    - Add tier recommendations: Baseline — plan migration timeline; Recommended — migrate to label groups, test Copilot DLP policies against new structure; Regulated — complete migration with validation that all DLP policies function correctly with label groups before moving to production.

    **Copilot Studio agent label inheritance:**
    - Document that Copilot Studio agents inherit the **highest sensitivity label** from their knowledge sources. If an agent's knowledge includes files labeled "Confidential" and "Public," the agent inherits "Confidential."
    - This has direct Copilot governance impact: agents with inherited high-sensitivity labels trigger label-based DLP policies when users interact with them.
    - Add tier recommendations: Baseline — audit agent knowledge source labels; Recommended — enforce label inheritance review as part of agent deployment approval; Regulated — mandatory label inheritance assessment with compliance sign-off before agent activation.

    **Auto-labeling nested rule logic:**
    - Document the new auto-labeling capability supporting nested AND/OR/NOT conditions (GA December 2025).
